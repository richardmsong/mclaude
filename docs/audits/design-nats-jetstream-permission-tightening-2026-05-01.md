1. **JWT revocation mechanism unimplemented and architecturally blocked** — The ADR references "CP adds the JWT to the NATS revocation list" in multiple critical flows (host access change propagation, emergency credential revocation, `manage.revoke-access`), but does not specify the NATS-level mechanism. Worse, the existing codebase (`mclaude-control-plane/admin.go:235-260`) documents that this is currently **impossible** with the project's infrastructure: (a) NATS account-level revocation requires re-signing the account JWT with the **operator key**, but CP only holds the account signing key (`NATS_ACCOUNT_SEED`); (b) pushing updated claims at runtime requires `$SYS.REQ.CLAIMS.UPDATE` with system account credentials, which CP does not have; (c) the hub NATS uses `resolver: MEMORY` with a preloaded account JWT, which does not support runtime claim updates without system account access. The ADR must either specify that CP loads the operator seed from `OPERATOR_KEYS_PATH` and obtains system account credentials, or switch to `resolver: nats` (full resolver), or accept that revocation will not work and redesign the affected flows to rely solely on short TTLs + proactive refresh. This is load-bearing: the host access grant/revoke flow says "NATS server: Closes bob's SPA WebSocket (revoked JWT)" — without working revocation, the old JWT remains valid for up to 8 hours (user JWT TTL).
   - **Doc**: "On host access change, CP revokes affected user JWTs and the SPA reconnects with a new JWT reflecting the updated host list." (Decisions table, "KV buckets" row); "CP adds the host JWT to the NATS revocation list (immediate disconnect)" (Credential Revocation section)
   - **Code**: `mclaude-control-plane/admin.go:235-260` — existing TODO block documents the architectural barrier: "NATS account-level revocation requires modifying the account JWT's Revocations list and re-signing it with the operator key. The control-plane only has the account signing key (NATS_ACCOUNT_SEED), not the operator key needed to re-sign the account JWT."

2. **Fan-out project creation — delivery pattern and error feedback unspecified** — The ADR changes project creation from the current HTTP-to-CP-to-NATS-request/reply flow (ADR-0035) to a fan-out model: "Users publish project create/delete to `mclaude.hosts.{hslug}.users.{uslug}.projects.{pslug}.{create,delete}`. However: (a) it's unclear whether the SPA publishes directly to NATS or continues using the existing HTTP `POST /api/users/{uslug}/projects` endpoint (which would then internally fan-out); (b) the error feedback mechanism is not specified — if CP rejects the project (slug collision, invalid host, authorization failure), how does the user learn about the error? NATS fan-out (pub/sub) does not support request/reply to multiple subscribers; (c) the `mclaude-web` Component Changes section does not mention any project creation flow change, suggesting the existing HTTP path might still be the intended SPA interface. A developer implementing this needs to know the exact SPA → CP → host-controller interaction pattern.
   - **Doc**: "No relay or interceptor needed — CP creates Postgres records and writes project KV; host controller starts/stops the agent subprocess." (BYOH Host section)
   - **Code**: `mclaude-control-plane/spec-control-plane.md` still documents `POST /api/users/{uslug}/projects` HTTP endpoint with no mention of deprecation or NATS migration

3. **Challenge nonce storage strategy unspecified** — The HTTP challenge-response authentication endpoints (`POST /api/auth/challenge` + `POST /api/auth/verify`) use single-use nonces with 30-second expiry. The ADR does not specify where nonces are stored (in-memory map, Postgres, external cache like Redis). For a single-replica control-plane deployment this is trivially in-memory, but the ADR doesn't state this assumption. If CP is scaled to multiple replicas, nonces generated by one replica would not be verifiable by another — the challenge would fail. The deployment model (single vs multi-replica) should be documented, or a shared nonce store specified.
   - **Doc**: "The challenge nonce is single-use and expires after 30 seconds." (Unified credential protocol section)
   - **Code**: CP is deployed as "a Kubernetes Deployment" (spec-control-plane.md) — Deployments support arbitrary replica counts

4. **`users.nkey_public` column missing UNIQUE constraint and cross-table lookup order** — The HTTP challenge endpoint searches for a public key across three tables: `users.nkey_public`, `hosts.public_key`, and `agent_credentials.nkey_public`. The `agent_credentials` table specifies `nkey_public TEXT NOT NULL UNIQUE`, but the `users` table change only says "new `nkey_public TEXT` column" with no uniqueness constraint. The `hosts.public_key` column (already exists) also has no documented uniqueness constraint. Without UNIQUE on `users.nkey_public`, two users could theoretically have the same public key (though cryptographically infeasible), and lookup performance degrades to a sequential scan. More critically, the lookup order across the three tables is not specified — a developer needs to know which table to query first to determine the identity type (user, host, or agent).
   - **Doc**: "CP looks up the public key in Postgres, determines the identity type (user, host, or agent)" (Unified credential protocol section)
   - **Code**: `agent_credentials` DDL shows `nkey_public TEXT NOT NULL UNIQUE`; `users` DDL shows only `nkey_public TEXT`

5. **Session subject verb tokens not in slug blocklist** — The consolidated session subject hierarchy uses `sessions.create` as a bare command verb (one token after `sessions.`) alongside session-scoped subjects `sessions.{sslug}.{operation}` (two+ tokens). The slug charset `[a-z0-9][a-z0-9-]*` permits slugs like `create`, `delete`, `input`, `events`, `config`, `control`, `lifecycle`. The ADR-0024 reserved-word blocklist (`users, hosts, projects, sessions, clusters, api, events, lifecycle, quota, terminal`) does NOT include `create`, `delete`, `input`, or `config`. While disambiguation by token count is structurally possible (bare verb = 1 token; slug+operation = 2+ tokens), this rule is implicit and undocumented. A session with slug `create` would produce subject `sessions.create` that is syntactically identical to the create command — the agent must parse token counts to distinguish them. The ADR should either add these verb names to the session slug blocklist or document the disambiguation rule.
   - **Doc**: Session subject hierarchy table shows `sessions.create` (verb, no slug) and `sessions.{sslug}.events` (slug + operation)
   - **Code**: `mclaude-common/pkg/slug` blocklist: `users, hosts, projects, sessions, clusters, api, events, lifecycle, quota, terminal` — does not include `create`, `delete`, `input`, `config`, `control`
