# ADR: Binary Data ‚Äî Imports and Attachments

**Status**: draft
**Status history**:
- 2026-04-29: draft (session import only)
- 2026-04-30: expanded to cover attachments, replaced NATS Object Store with S3 + pre-signed URLs

## Overview

Unified binary data architecture for mclaude. All binary data ‚Äî session imports, chat attachments (images, screenshots, files), and any future binary payloads ‚Äî flows through S3-compatible object storage with pre-signed URLs. NATS carries lightweight references, never bytes.

This ADR covers two features:
1. **Session import**: CLI command (`mclaude import`) that imports existing Claude Code session data into mclaude.
2. **Attachments**: Images, screenshots, and files attached to chat messages by users or generated by agents.

Both use the same storage path: S3 upload via pre-signed URL, S3 download via pre-signed URL, NATS messages carry references only.

## Motivation

**Imports:** Users working with Claude Code locally accumulate valuable session history (JSONL transcripts with full conversation context, tool usage, and reasoning) and project memories. When they want to bring this work into mclaude ‚Äî to continue sessions via the mclaude platform, share context with teammates, or benefit from mclaude's session management ‚Äî there is currently no mechanism to do so.

**Attachments:** Users need to share screenshots, images, error logs, and other files with session agents during chat. Agents may generate images, diagrams, or other binary output. These must not flow through NATS (1MB message limit, bloats stream replay, wastes bandwidth on every `DeliverAll`).

**Why not NATS Object Store:** The original design used per-user NATS Object Store buckets (`OBJ_mclaude-imports-{uslug}`). Each bucket is a JetStream stream. Attachments would require per-project buckets (to match the agent JWT security boundary), creating O(users √ó projects) streams ‚Äî doesn't scale. Object Store also has no per-object permissions (chunk IDs are opaque, can't be scoped in JWTs), requiring one-shot JWTs for download authorization. S3 + pre-signed URLs eliminates all of this: time-limited, scoped access tokens are built into the URL itself. No extra JetStream streams, no one-shot JWTs, no chunk enumeration.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Import surface | `mclaude` CLI subcommand | Portable, scriptable, doesn't require Factory session. A skill wrapper can be added later. |
| Binary data storage | S3-compatible object storage (MinIO for self-hosted, AWS S3 for cloud) | Scalable blob storage with built-in pre-signed URL support. No NATS Object Store ‚Äî eliminates per-user/per-project JetStream stream proliferation. |
| Upload/download mechanism | Pre-signed URLs (client ‚Üî S3 directly, CP only signs) | CP never proxies binary data. Handles auth via URL signing (time-limited, scoped). Both uploads and downloads go directly to/from S3. |
| NATS role for binary data | References only (`AttachmentRef` with opaque ID) | NATS messages stay lightweight. No 1MB limit concerns. Stream replay doesn't re-transfer binary data. |
| Session selection | All sessions for the CWD | Import is a bulk operation for all historical sessions. No per-session selection in v1. |
| Post-import behavior | Imported as read-only history; resume is a future enhancement | v1 focuses on getting data into mclaude. Resume requires session-agent `--resume` plumbing ‚Äî deferred. |
| Archive format | tar.gz | Standard, widely supported, handles directory trees naturally |
| Project creation | Import always creates a NEW project | No merging into existing projects ‚Äî merging risks conflicts. If name collides, user must rename. |
| Auth | User-level JWT via `mclaude login` (device-code flow) | Import acts on behalf of the user, not the host. Same NATS JWT mechanism as SPA login. |
| Storage placement | Controller-dependent (harness-aware) | BYOH/local: `~/.claude/projects/` native location. K8s: PVC mounted by session-agent. Controller decides. |
| K8s scope | In scope for v1 | Both BYOH/local and K8s deployment targets supported. |
| Upload security | Pre-signed URLs with scoped S3 keys | Each pre-signed URL grants access to exactly one object for a limited time (5 min). No broad bucket access. CP validates project/user ownership before signing. |
| Attachment scoping | Per-project S3 key prefix | S3 keys use `{uslug}/{hslug}/{pslug}/attachments/{id}` ‚Äî CP validates the requesting identity owns that project before signing a URL. A compromised agent credential only gets URLs for its own project. |
| Session IDs | Preserve original Claude Code session IDs | Enables future `--resume`. Re-import of same session ID: session-agent skips the duplicate session with a warning and imports the remaining sessions. |
| Name collision | Prompt user for new name | No auto-suffixing. User has full control over project naming. |
| UI distinction | None ‚Äî imported sessions treated same as native | No "imported" badge. Simplifies UI. Metadata stored internally but not surfaced. |

## User Flow ‚Äî Import

1. User runs `mclaude login` (if not already authenticated) ‚Äî device-code flow ‚Üí credentials stored at `~/.mclaude/auth.json` (`{ jwt, nkeySeed, userSlug }`)
2. User runs `mclaude import [--host <hslug>]` from their project directory
3. CLI derives encoded CWD (absolute path with `/` ‚Üí `-`, leading `-` stripped; e.g. `/Users/rsong/work/mclaude` ‚Üí `Users-rsong-work-mclaude`), discovers session data at `~/.claude/projects/{encoded-cwd}/`:
   - JSONL transcripts: `{sessionId}.jsonl`
   - Subagent data: `{sessionId}/subagents/`
   - Memories: `memory/` directory
   - Project CLAUDE.md (from CWD `.claude/CLAUDE.md` or `CLAUDE.md`)
4. CLI connects to NATS using stored JWT + NKey seed
5. CLI derives project name from directory name (last path component). Checks slug availability via NATS request/reply to control-plane (`mclaude.users.{uslug}.hosts.{hslug}.projects.check-slug`). Slug uniqueness is per-user-per-host, so the host token is required in the subject.
6. If slug taken ‚Üí CLI prompts user for a new name ‚Üí re-checks. Loop until available.
7. CLI packages data into tar.gz with metadata.json
8. CLI requests pre-signed upload URL from CP via NATS request/reply (`mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.import.request`)
9. CLI uploads archive directly to S3 using the signed URL
10. CLI signals CP that upload is complete via NATS request/reply (`mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.import.confirm`)
11. Control-plane creates new project (Postgres + KV) with `importRef` in state (import ID, e.g. `imp-001`), dispatches provisioning to controller
12. Controller creates pod/process; session-agent starts
13. Session-agent reads project state, sees `importRef` (import ID). Requests pre-signed download URL from CP via NATS request/reply (`mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.import.download`)
14. Session-agent downloads archive directly from S3, unpacks to PVC (K8s) or `~/.claude/projects/{encoded-cwd}/` (local)
15. Session-agent clears `importRef` from project KV state, publishes `import.complete` via NATS (`mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.import.complete`) with the standard message envelope payload `{"id":"...","ts":...}` (no additional fields ‚Äî the subject encodes user/host/project, and CP resolves the import ID from project KV state); CP receives signal and deletes the S3 object
16. Session-agent's fsnotify watcher detects new JSONL files, creates SessionState KV entries
17. Sessions appear in web UI under the new project (read-only history; resume is a future enhancement)

**CLI bootstrapping:** The CLI reads the control-plane server URL from `~/.mclaude/context.json`'s `server` key (consistent with the existing CLI spec). The `--server <url>` flag overrides this value. Default: `https://api.mclaude.internal`.

## User Flow ‚Äî Attachments

### Upload (user sends screenshot in chat)

1. SPA requests pre-signed upload URL from CP: `POST /api/attachments/upload-url {filename, mimeType, sizeBytes, projectSlug, hostSlug}`
2. CP validates user owns the project, generates S3 key (`{uslug}/{hslug}/{pslug}/attachments/{id}`), signs upload URL
3. CP returns `{id: "att-001", uploadUrl: "https://s3.../...?X-Amz-Signature=..."}`
4. SPA uploads file directly to S3 using the signed URL
5. SPA confirms upload: `POST /api/attachments/{id}/confirm`
6. SPA sends chat message via NATS with `AttachmentRef`:
   ```json
   {"type":"message","text":"what's wrong here?","attachments":[{"id":"att-001","filename":"screenshot.png","mimeType":"image/png","sizeBytes":245000}]}
   ```

### Download (agent needs the screenshot)

1. Agent receives the session input message containing `AttachmentRef`
2. Agent requests pre-signed download URL from CP via NATS request/reply (`mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.attachments.download`), sending `{id: "att-001"}` ‚Äî CP returns pre-signed URL
3. CP validates agent's project ownership (agent JWT contains uslug/hslug/pslug; attachment S3 key matches)
4. CP returns `{downloadUrl: "https://s3.../...?X-Amz-Signature=..."}`
5. Agent downloads file directly from S3

### Agent-generated attachments

Same flow but reversed:
1. Agent requests upload URL from CP via NATS request/reply (`mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.attachments.upload`), sending `{filename, mimeType, sizeBytes}` ‚Äî CP returns `{id, uploadUrl}`
2. Agent uploads to S3, confirms via NATS request/reply (`mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.attachments.confirm`), sending `{id: "att-002"}`
3. Agent publishes session event with `AttachmentRef` in content blocks
4. SPA receives event, requests download URL from CP, renders the image/file

## Component Changes

### mclaude-cli
- Updated `mclaude login` subcommand (extends existing command with NATS credential support via device-code flow):
  - Initiates device-code flow: `POST /api/auth/device-code` to control-plane ‚Üí returns `{ deviceCode, userCode, verificationUrl, expiresIn, interval }`
  - Displays URL + code to user, polls `POST /api/auth/device-code/poll` until completion
  - Receives `{ jwt, userSlug }` ‚Äî the CLI generates its own NKey pair locally. It sends only the public key in the device-code request. The poll response returns the signed JWT and user slug ‚Äî no seed, no NKey material from the server.
  - Stores all credentials at `~/.mclaude/auth.json` (same path as before; credential format changes from bearer token to `{ jwt, nkeySeed, userSlug }` ‚Äî NATS JWT + NKey seed + user slug in a single file)
  - JWT refresh: before import operations, check TTL ‚Üí `POST /auth/refresh` if needed
- New `mclaude import` subcommand:
  - Loads auth from `~/.mclaude/auth.json` (error if not logged in)
  - Reads active host from context (`~/.mclaude/context.json`); `--host <hslug>` overrides
  - Derives encoded CWD from current directory using Claude Code's path encoding algorithm: take the absolute path (e.g. `/Users/rsong/work/mclaude`), replace every `/` with `-`, strip the leading `-` (e.g. `Users-rsong-work-mclaude`). This matches the directory names under `~/.claude/projects/`. The CLI must verify at runtime that the derived path exists under `~/.claude/projects/`; if it doesn't, the CLI should list available encoded directories and error with a hint
  - Discovers session data at `~/.claude/projects/{encoded-cwd}/`
  - Derives project name from last path component of CWD; checks slug uniqueness via control-plane
  - If slug collision: prompts user for new name
  - Packages data into `import-{slug}.tar.gz` with `metadata.json`
  - Requests pre-signed upload URL via NATS request/reply (`import.request` subject): `{slug, sizeBytes}`
  - Uploads archive directly to S3 using the signed URL
  - Confirms upload via NATS request/reply (`import.confirm` subject): `{"importId":"imp-001"}` (the import ID returned by `import.request`; `hostSlug` is already encoded in the subject)
  - Waits for provisioning acknowledgement, prints success message

### mclaude-control-plane
- New device-code auth endpoints:
  - `POST /api/auth/device-code` ‚Äî generate device code, store pending auth in memory with TTL
  - `POST /api/auth/device-code/poll` ‚Äî CLI polls with device code, returns credentials when user completes auth
  - `GET /api/auth/device-code/verify` ‚Äî web UI endpoint where user enters the code and authenticates
- New S3 integration:
  - S3 client configured via environment (`S3_ENDPOINT`, `S3_BUCKET`, `S3_ACCESS_KEY_ID`, `S3_SECRET_ACCESS_KEY`)
  - Single bucket with key prefix per user/host/project: `{uslug}/{hslug}/{pslug}/imports/{id}` and `{uslug}/{hslug}/{pslug}/attachments/{id}`
  - Pre-signed URL generation with configurable TTL (default: 5 min for uploads, 5 min for downloads)
- New import NATS request/reply handlers (CLI-to-CP and agent-to-CP operations use NATS, not HTTP):
  - `import.request` (subject: `mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.import.request`) ‚Äî validates user auth, generates S3 key + pre-signed upload URL
  - `import.confirm` (subject: `mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.import.confirm`) ‚Äî validates upload exists in S3, creates project in Postgres with `source: "import"` and `import_ref` (S3 key), writes `ProjectKVState` with `importRef` (import ID), dispatches provisioning to controller
  - `import.download` (subject: `mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.import.download`) ‚Äî validates requester (agent) owns the project, returns pre-signed download URL
  - Note: S3 object deletion is triggered by the agent's `import.complete` NATS publication (no HTTP DELETE endpoint needed for agent-to-CP)
- New attachment HTTP endpoints (SPA-to-CP):
  - `POST /api/attachments/upload-url` ‚Äî validates user owns the project, generates S3 key + pre-signed upload URL
  - `POST /api/attachments/{id}/confirm` ‚Äî records attachment metadata in Postgres (id, filename, mimeType, sizeBytes, s3Key, project, user)
  - `GET /api/attachments/{id}` ‚Äî validates requester owns the project, returns pre-signed download URL
- New attachment NATS request/reply handlers (agent-to-CP):
  - `attachments.download` (subject: `mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.attachments.download`) ‚Äî agent sends `{id}`, CP validates project ownership, returns `{downloadUrl}`
  - `attachments.upload` (subject: `mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.attachments.upload`) ‚Äî agent sends `{filename, mimeType, sizeBytes}`, CP returns `{id, uploadUrl}`
  - `attachments.confirm` (subject: `mclaude.users.{uslug}.hosts.{hslug}.projects.{pslug}.attachments.confirm`) ‚Äî agent sends `{id}`, CP records metadata in Postgres
- Attachment metadata table in Postgres:

### Postgres: `attachments`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | TEXT | PRIMARY KEY | Opaque attachment ID (e.g. `att-001`), generated by CP on `upload-url` request |
| `s3_key` | TEXT | UNIQUE NOT NULL | Full S3 object key (`{uslug}/{hslug}/{pslug}/attachments/{id}`) |

| `project_id` | TEXT | NOT NULL FK‚Üíprojects ON DELETE CASCADE | Project the attachment belongs to |
| `confirmed` | BOOLEAN | NOT NULL DEFAULT FALSE | TRUE after `POST /api/attachments/{id}/confirm` (unconfirmed rows are garbage-collected) |

Indexes:
- `idx_attachments_project` on `(project_id)` ‚Äî lookup for project-level cleanup on project deletion

Writers: control-plane (upload-url handler creates row with `confirmed=false`; confirm handler sets `confirmed=true`)
Readers: control-plane (download handler, project deletion cleanup)

### mclaude-session-agent
- New import handler:
  - On startup, reads project KV state
  - If `importRef` is set: requests pre-signed download URL from CP via NATS request/reply (`import.download`), downloads archive directly from S3, unpacks to session data directory
  - After successful unpack: clears `importRef` from KV state, publishes `import.complete` via NATS with standard envelope payload `{"id":"...","ts":...}` (CP receives signal, resolves import ID from project KV, and deletes S3 object)
  - Reports import completion via lifecycle event
- New attachment support:
  - When processing a user input with `AttachmentRef`: requests download URL from CP via NATS request/reply (`attachments.download` subject, sends `{id}`), downloads from S3, passes to CLI driver as image/file input
  - When agent generates binary output: requests upload URL from CP via NATS request/reply (`attachments.upload` subject, sends `{filename, mimeType, sizeBytes}`), uploads to S3, confirms via `attachments.confirm` subject, includes `AttachmentRef` in session event
- New fsnotify watcher:
  - Watches session data directory for new `.jsonl` files
  - When new JSONL file detected: extracts session metadata (ID, timestamps, branch, model from first few lines)
  - Creates `SessionState` KV entry with state `"completed"` (imported sessions are historical)
  - Handles both import-triggered discovery and any future file-based session discovery

### mclaude-common
- New types: `ImportRequest`, `ImportMetadata`, `AttachmentRef` (id, filename, mimeType, sizeBytes), `AttachmentMeta` (Postgres row: id, s3Key, filename, mimeType, sizeBytes, userSlug, hostSlug, projectSlug, createdAt)
- Extended `ProjectKVState` / `ProjectState` with `importRef` field (import ID, e.g. `imp-001`; replaces `ImportObjectRef`). CP stores the S3 key separately in the Postgres `import_ref` column.
- Extended `ProjectState` Postgres model with `source` and `import_ref` columns

### mclaude-web
- New device-code verification page: `/auth/device-code/verify` ‚Äî user enters code, authenticates, redirected to success screen
- Attachment upload in chat input: file picker / paste handler ‚Üí request upload URL ‚Üí upload to S3 ‚Üí confirm ‚Üí attach `AttachmentRef` to input message
- Attachment rendering in chat: resolve `AttachmentRef` in session events ‚Üí request download URL ‚Üí render image/file preview
- No changes to session list or project views (imported sessions treated identically to native)

## Error Handling

| Error | Component | Behavior |
|-------|-----------|----------|
| Not logged in | CLI | `mclaude login` required first. Clear error message. |
| No session data found for CWD | CLI | Error listing expected path `~/.claude/projects/{encoded-cwd}/` |
| No active host in context | CLI | Error: `mclaude host register` and `mclaude host use` required first |
| Session JSONL files corrupted/truncated | CLI | Warn per file but include what's available in archive |
| Project slug collision | CLI + Control-plane | Control-plane returns 409. CLI prompts user for new name. |
| Session ID collision (re-import) | Session-agent | Skip the conflicting session with a warning, import remaining sessions |
| NATS connection failed | CLI | Retry with backoff, clear error with NATS URL and troubleshooting hints |
| S3 upload failed mid-stream | CLI | Retry from beginning (S3 multipart uploads handle this natively) |
| Pre-signed URL expired | CLI/Agent | Request a new URL from CP, retry |
| Archive too large | CLI | Pre-check archive size against configurable limit (default 500MB), warn before upload |
| Attachment too large | SPA/Agent | CP rejects upload-url request if sizeBytes exceeds limit (default 50MB) |
| Unpack fails (disk full, permissions) | Session-agent | Report error via lifecycle event, leave `importRef` set for retry |
| Device-code expired | CLI + Control-plane | 15-minute TTL. CLI displays "code expired, try again" |
| Device-code not yet verified | CLI polling | Continue polling until verified or expired |

## S3 Key Structure

Single S3 bucket, all data organized by user/host/project prefix:

```
{uslug}/{hslug}/{pslug}/imports/{import-id}.tar.gz
{uslug}/{hslug}/{pslug}/attachments/{attachment-id}
```

CP validates ownership (user JWT or agent JWT contains uslug/hslug/pslug) before signing any URL. Pre-signed URLs grant access to exactly one object for a limited time (5 min). No broad bucket listing or prefix access.

**Cleanup:**
- Import archives: deleted by CP when agent signals unpack complete
- Attachments: retained as long as the project exists. On project deletion, CP deletes the S3 prefix `{uslug}/{hslug}/{pslug}/`

## Security

- Archive contains full conversation history including potentially sensitive tool outputs ‚Äî S3 uploads/downloads use HTTPS (TLS in transit), S3 encryption at rest (SSE-S3 or SSE-KMS)
- Memories may contain user-specific context ‚Äî ownership enforced by CP at URL-signing time
- Pre-signed URLs are time-limited (5 min) and scoped to a single S3 object ‚Äî no broad bucket access
- S3 keys encode the project path (`{uslug}/{hslug}/{pslug}/...`) ‚Äî CP checks this against the requester's identity before signing
- A compromised agent can only get URLs for its own project (CP validates uslug/hslug/pslug from the agent JWT)
- Session-agent validates archive integrity (metadata.json schema, JSONL line format) before unpacking
- CLI validates file sizes before packaging (guard against accidentally archiving node_modules etc.)
- Device-code auth: codes expire after 15 minutes, single-use, rate-limited
- Attachment upload size limits enforced by CP when generating upload URLs (configurable, default 50MB per file)

## Dependencies

- **ADR-0054 (NATS JetStream Permission Tightening)** ‚Äî Required for NATS credential model. Binary data no longer flows through NATS Object Store, but the NATS permission model (user/agent JWT scoping) is still needed for the rest of the system.
- **S3-compatible storage** ‚Äî MinIO for self-hosted deployments, AWS S3 for cloud. Must support pre-signed URLs (all S3-compatible implementations do).

## Scope

**v1:**
- S3 integration in control-plane (MinIO for dev/self-hosted, S3 for cloud)
- `mclaude login` ‚Äî device-code auth flow for CLI
- `mclaude import` ‚Äî discover, package, upload session data via S3 pre-signed URLs
- Control-plane import endpoints ‚Äî upload-url, confirm, download-url, delete
- Control-plane attachment endpoints ‚Äî upload-url, confirm, download
- Session-agent import handler ‚Äî S3 download, unpack, advertise sessions
- Session-agent attachment support ‚Äî download user attachments for CLI input, upload agent-generated attachments
- SPA attachment support ‚Äî file picker, paste, upload, rendering
- Session-agent fsnotify watcher ‚Äî live session discovery from JSONL files
- Web device-code verification page
- Both BYOH/local and K8s deployment targets

**Deferred:**
- Session resume (`--resume` from imported JSONL) ‚Äî future enhancement
- Incremental sync (only new sessions since last import)
- Bidirectional sync
- Import from remote machines
- Import specific session subsets (date range, etc.)
- Import progress streaming (real-time progress bar in CLI)
- Archive compression options (different algorithms, size limits)
- Attachment thumbnails/previews (generate server-side for faster SPA rendering)
- Attachment expiry/lifecycle management (auto-delete old attachments)
