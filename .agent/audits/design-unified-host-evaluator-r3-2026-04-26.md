3. **Device-code status-check endpoint for CLI polling is undefined** — User Flow A step 4 states "CLI receives the response (poll-based: device-code endpoint exposes a status check)". No such poll endpoint is defined in spec-control-plane.md or the ADR. spec-control-plane.md defines only `POST /api/users/{uslug}/hosts/code` (generate) and `POST /api/hosts/register` (redeem by dashboard). The CLI implementer cannot build the `mclaude host register` polling loop without knowing what endpoint to poll or what response to expect.
   - **Doc**: `docs/adr-0035-unified-host-architecture.md` line 63: "CLI receives the response (poll-based: device-code endpoint exposes a status check)"
   - **Code**: `docs/mclaude-control-plane/spec-control-plane.md` protected endpoints table — no GET endpoint for device-code status.

6. **User Flow B step 2 says "generates the cluster's account JWT" but there is exactly one account per install** — The NATS auth chain Decisions row (line 44) and spec-state-schema.md (line 506) both state there is one operator and one account per mclaude install, and the worker NATS uses the same account JWT as the hub.    - **Doc**: `docs/adr-0035-unified-host-architecture.md` line 70: "It also generates the cluster's account JWT signed by the deployment-level account key."
