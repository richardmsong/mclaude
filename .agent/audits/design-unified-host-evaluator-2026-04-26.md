3. **`$SYS` presence mapping — NKey-to-host resolution unspecified** — The design says control-plane maps "account-id-to-host-slug via the host's NKey public key". With a single account per install, `$SYS.ACCOUNT.{accountKey}.CONNECT` uses the **one** account's public key for every connect event; the per-client identity is in the payload, not the subject. The doc does not specify: (a) which field in the `$SYS.ACCOUNT.*.CONNECT` payload carries the per-user NKey public key, (b) how control-plane queries Postgres to resolve `public_key → (user_id, host_id)`, or (c) what happens when the connecting client is the SPA (which does not have a `hosts.public_key` row). An implementer cannot write the presence handler without knowing the payload field name and the lookup query.
   - **Doc**: ADR-0035 Component Changes, control-plane: "Subscribe to `$SYS.ACCOUNT.{accountKey}.CONNECT/DISCONNECT` on hub NATS; map account-id-to-host-slug via the host's NKey public key"; spec-state-schema NATS Subjects table: `$SYS.ACCOUNT.{accountKey}.CONNECT` — "control-plane maps NKey public key → host slug"
   - **Code**: `mclaude-control-plane/nkeys.go` — no `$SYS` subscription logic exists; no presence handler.

5. **Leaf-node remote auth format conflict: `nkey` vs `credentials`** — ADR-0035 Component Changes section (mclaude-helm) and the `charts/mclaude-worker/` template description show the worker NATS leaf-node remote configured as `nkey: /etc/nats/leaf.nk` (raw NKey seed file). `spec-state-schema.md` NATS Server Configuration for Worker NATS shows `credentials: /etc/nats/leaf.creds` (standard `.creds` file containing JWT + seed). These are different NATS config directives: `nkey` uses NKey challenge-response without a JWT, while `credentials` uses a `.creds`-format file requiring a JWT. The decision "NATS auth chain: 3-tier operator → account → user JWT" implies the leaf-node connection must use a JWT, making `credentials` correct — but the mismatch means a developer implementing the Helm chart cannot determine which config key and file format to use.
   - **Doc**: ADR-0035 Component Changes (mclaude-helm): `leafnodes { remotes: [{url: nats-leaf://hub:7422, nkey: /etc/nats/leaf.nk}] }`; spec-state-schema Worker NATS config: `credentials: /etc/nats/leaf.creds`
   - **Code**: No chart code exists yet; the two specs that describe the same config block disagree on the directive name.

7. **BYOH device-code registration: NKey generated by host or control-plane is contradicted within the same ADR** — `spec-control-plane.md` (the updated spec this ADR claims to create) defines `POST /api/hosts/register` as receiving `{code, name, publicKey}` from the CLI (host-generated NKey, private seed never leaves the machine — consistent with ADR-0004 security decision).    - **Doc**: ADR-0035 User Flow A step 3: "control-plane ... generates an NKey pair ... returns `{slug, jwt, nkeySeed, hubUrl}`"; spec-control-plane.md line 63: `POST /api/hosts/register` receives `{code, name, publicKey}` and returns `{slug, jwt, hubUrl}` (no `nkeySeed`).
