The canonical `subj` package only exposes `UserHostProjectAPISessionsCreate(u, h, p)`, `UserHostProjectAPISessionsInput(u, h, p)`, `UserHostProjectAPISessionsDelete(u, h, p)`, and `UserHostProjectLifecycle(u, h, p, s)` — all taking a `slug.HostSlug` parameter. The existing code in `daemon_jobs.go` calls non-existent two-argument variants (`subj.UserProjectAPISessionsCreate`, `subj.UserProjectAPISessionsInput`, `subj.UserProjectAPISessionsDelete`, `subj.UserProjectLifecycle`). The design doc never mentions how the host slug will be obtained, stored on `JobEntry`, or passed to these functions. Without this, all NATS publishes go to subjects that no session-agent subscribes to.
   - **Doc**: "Dispatcher calls `sessions.create` with `permPolicy`, `allowedTools`, and the worktree branch." / "Sends the nudge via `sessions.input` (same session ID; same live subprocess)." — no mention of host slug in any call.
