1. **Plugin hook registration mechanism is wrong** — The ADR says the setup skill "registers the pre-tool-use hook in project `.claude/settings.json`" and specifies the plugin ships `hooks/pre-tool-use.sh`. This is incorrect on both counts. Actual Claude Code plugin hooks are declared in `hooks/hooks.json` (a JSON file with a `{"description": ..., "hooks": {...}}` wrapper) and are active for all sessions using the plugin — not registered per-project via settings.json. The setup skill approach of modifying project settings.json to point at a hook script contradicts how plugins actually deliver hooks. A developer implementing this would not know: (a) that `hooks/hooks.json` is required rather than a bare `.sh` file, (b) what the per-project opt-in mechanism should be (the correct pattern per the `plugin-dev` plugin docs is a flag file checked at runtime: `if [ ! -f "$CLAUDE_PROJECT_DIR/.agent/blocked-commands.json" ]; then exit 0; fi`).
   - **Doc**: "registers pre-tool-use hook in project `.claude/settings.json`" and "hooks/ └── pre-tool-use.sh"
   - **Code**: `~/.claude/plugins/marketplaces/claude-plugins-official/plugins/learning-output-style/hooks/hooks.json` shows the actual format: `{"description": "...", "hooks": {"SessionStart": [...]}}`. The `plugin-dev` skill at `skills/hook-development/SKILL.md` explicitly states: "For plugin hooks in `hooks/hooks.json`, use wrapper format" and the hook script is referenced from within that JSON via `"command": "bash \"${CLAUDE_PLUGIN_ROOT}/hooks-handlers/session-start.sh\""`.

2. **Plugin `.mcp.json` format is unspecified/ambiguous** — The ADR's proposed `.mcp.json` is `{ "mcpServers": { "docs": { "command": "docs-mcp", "args": ["--root"] } } }`. The `"mcpServers"` wrapper may or may not be correct for a dedicated plugin `.mcp.json`. The official `plugin-dev` plugin's `mcp-integration` skill (Method 1 — dedicated `.mcp.json`, recommended) shows flat keys without the wrapper, while `plugin-structure` skill shows the wrapper format. A developer implementing this would need to guess which format Claude Code actually parses for plugin-scoped `.mcp.json`. Additionally, the proposed command `"docs-mcp"` relies on the binary being on PATH (via the `~/.local/bin` symlink), but plugin MCP configs canonically use `"${CLAUDE_PLUGIN_ROOT}/bin/docs-mcp"` for portability. The `--root` arg with no value is described as "tells docs-mcp to use the walk-up-to-.git fallback" but this is a non-standard CLI convention that needs explicit implementation in `index.ts` and is not documented there.
   - **Doc**: `"command": "docs-mcp"` and `"args": ["--root"]` with no value; the comment "The `--root` arg with no value tells docs-mcp to use the walk-up-to-.git fallback"

No existing plugin uses a bare binary name without `${CLAUDE_PLUGIN_ROOT}` prefix. `mclaude-docs-mcp/src/index.ts` has no `--root` arg parsing at all.

3. **Plugin installation flow is wrong** — The ADR says: "Install plugin: `claude plugin install <repo-url>` or local path". The actual `claude plugin install` command only installs from a named marketplace plugin, not from a URL or path. The correct flow for a standalone repo is: `claude plugin marketplace add <path-or-url>` first, then `claude plugin install spec-driven-dev`. A developer following the ADR's installation instructions would fail on step 1.
   - **Doc**: "Install plugin: `claude plugin install <repo-url>` or local path"
   - **Code**: `claude plugin install --help` shows `<plugin>` argument is a marketplace plugin name. `claude plugin marketplace add --help` shows "Add a marketplace from a URL, path, or GitHub repo".
